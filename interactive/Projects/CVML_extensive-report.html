Ignition Server — The OT/IT Integration Hub
The Ignition SCADA server acted as the central integration hub by hosting both Mosquitto (OT-side MQTT) and RabbitMQ (IT-side AMQP) alongside Ignition SCADA itself. This deliberately co-located architecture made Ignition the explicit bridge between device-level OT messaging and enterprise-level IT messaging — translating raw inspection events into structured quality records. One server. Two brokers. One integration boundary. A deliberate simplification for 2020–21 — and the primary single point of failure.
RPi Capture → MQTT Publish → Mosquitto → MQTT Bridge → EC2 Subscriber → model.predict() → AMQP Publish → RabbitMQ → MES Write + SCADA Update
ML Model Lifecycle

Training Pipeline & Model Versioning

How the ResNet50 model was built, evaluated, versioned, and deployed to EC2 — from raw labeled images to live inference in 2020–2021.

Training Pipeline
1
Dataset Labeling
Quality engineers labeled ~85K images captured from production lines using Label Studio. 5 defect classes: pass, scratch, dimensional, foreign material, other. Images exported as JSON annotations linked to image paths in S3.
Label Studio → S3 JSON annotations
2
Data Preparation & Augmentation
Python script loads images from S3, splits into train/val/test (70/15/15). Augmentation applied: rotation ±15°, brightness jitter ±20%, Gaussian noise, horizontal flip — addresses class imbalance (FAIL images ~50% of PASS count).
TF ImageDataGenerator · class_weight balancing
3
Transfer Learning — ResNet50
ImageNet pre-trained ResNet50 base frozen. Custom classification head added: GlobalAveragePooling → Dense(256, ReLU) → Dropout(0.4) → Dense(5, Softmax). Phase 1: train head only (10 epochs). Phase 2: unfreeze last 30 layers, fine-tune (20 epochs, lr=1e-5).
TensorFlow 2.x · Keras · ResNet50 · GPU EC2
4
Model Evaluation & Acceptance Gate
Evaluated on held-out test set. Promotion criteria: overall accuracy ≥ 93%, FAIL class recall ≥ 91% (critical — missing FAILs is worse than false alarms). Confusion matrix reviewed by quality engineering lead before sign-off.
sklearn classification_report · confusion matrix
5
Model Artifact Storage — S3
Approved model saved as model_v{N}.h5 and uploaded to s3://cvml-manufacturing-2020/model-artifacts/v{N}/. Companion metadata JSON: accuracy, recall, training date, dataset hash, engineer sign-off name.
boto3 upload · model_v1.0.h5 · model_v1.1.h5
6
Deployment to EC2
EC2 inference process loads model on startup via tf.keras.models.load_model() from local disk. Updated model deployed by: (1) SSH to EC2, (2) boto3 download from S3, (3) restart Python inference process. Version string logged in every MES inspection record.
Manual SSH deploy · model_version logged in MES

Storage Gaps — Legacy System
  • ✕ No S3 lifecycle policy — ~65–200 GB/yr cost growth unmonitored
  • ✕ Non-atomic write — S3 upload and MES record written separately; failures left dangling URLs in MES
  • ✕ FAIL images not prioritised — no selective retention for retraining pipeline
  • ✕ EC2 local disk as unintended staging area — images stranded after crashes
  • ✕ No image validation — MES stored S3 URL without confirming object exists

Security Architecture

Security Layer — Legacy & Known Gaps

What security controls were in place in 2020–21, and where the architecture fell short against modern OT/IT security standards. Understanding the gaps is the foundation for the next-generation redesign.

🔐
Transport Security
MQTT + AMQP layer
  • TLS 1.2 on Mosquitto port 8883 — all RPi → Mosquitto traffic encrypted
  • TLS on Mosquitto bridge to AWS EC2 — Internet hop encrypted
  • Username/password auth on Mosquitto — shared credentials across all 6 RPi stations
  • RabbitMQ AMQP behind plant firewall — no external exposure
  • TLS on EC2 → RabbitMQ AMQP connection
⚠ Gap: shared credentials, no per-device identity. Any compromised RPi exposed all stations.
☁
AWS Cloud Security
EC2 + S3 layer
  • EC2 instance in default VPC — public subnet with security group rules
  • Security group: inbound MQTT port 1883/8883 from Mosquitto bridge IP only
  • IAM role attached to EC2 instance — S3 read/write permissions scoped to cvml bucket
  • S3 bucket policy: deny public access, EC2 role only
  • SSH access restricted to engineering VPN IP range only
⚠ Gap: EC2 in public subnet, no VPC private subnet isolation, no WAF. IAM role scoped but overly broad (full S3 bucket access).
🏭
Plant OT Network Security
Firewall + segmentation
  • Plant firewall allows outbound port 8883 only from Ignition Server IP
  • RPi devices on dedicated plant Wi-Fi SSID — isolated from corporate network
  • Ignition Server on OT VLAN — no direct route to corporate IT network
  • MES on separate VLAN — adapter accesses via stored procedure only
  • Physical access to RPi units controlled by plant floor access badge
⚠ Gap: no DMZ between Mosquitto and EC2. Direct bridge from OT network to public cloud IP. No network traffic monitoring or IDS on OT VLAN.

Quality Logic

PASS / FAIL Decision Flow

Eight sequential steps from photoelectric sensor trigger to MES inspection record write. Total elapsed time: 3.2–4.0 seconds.

01
Trigger
Sensor → GPIO
Photoelectric sensor detects part in position → GPIO pin goes HIGH → RPi Python interrupt fires camera capture routine.
02
Capture
Image + Preprocess
12MP capture → resize 224×224 → Gaussian blur (3×3) → histogram equalization → base64 encode → assemble JSON payload with station_id, part_serial, timestamp.
03
Transport
MQTT Publish
Paho-MQTT publishes to plant/{line}/stn{id}/inspection. QoS Level 1. Mosquitto receives on Ignition Server.
04
Bridge
Mosquitto → EC2
Mosquitto bridge plugin forwards message to EC2 Paho-MQTT subscriber over Internet (TLS). EC2 process is a persistent long-running subscriber — the key reason Lambda was not used.
05
Inference
ResNet50 ML Model
base64 decode → NumPy tensor → TF preprocessing → ResNet50 model.predict() → class label (PASS/FAIL) + confidence float (0.00–1.00). Largest single latency contributor: 1.0–1.5 sec on CPU.
06
Routing
Result → RabbitMQ
EC2 AMQP client publishes JSON result to RabbitMQ on Ignition Server. Routing key: {station_id}.{result}. MES adapter + Ignition both consume simultaneously.
07
MES Write
Adapter Consumer
Python MES adapter consumes AMQP queue → reads Work Order + Part Serial from MES → writes inspection record via stored procedure → ACK sent only after successful write.
08
SCADA
Ignition Update
Ignition MQTT Engine populates OPC-UA tags per station. Operator HMI updates with result + image. 3× consecutive FAILs triggers Line Stop alarm to supervisor.

✕ FAIL

Operational Resilience

Failure Modes & Fault Tolerance

Factories require explicit fault handling for every system component. Below is the complete failure mode matrix for the legacy CVML system — what was handled, what wasn't, and the consequences on the production floor.

Failure Mode Severity Production Impact Legacy Handling Gaps / What Was Missing
Plant Wi-Fi Drop MEDIUM Inspections queued but not sent. Parts hold at station until reconnect. ✓ SQLite local buffer on RPi (added v1.1). Images stored up to 10-min age gate, then replayed on reconnect. Not in v1.0 — data loss during Wi-Fi outages pre-patch. No alerting to supervisor that station was in offline mode.
EC2 Process Crash HIGH All inference halted. MQTT messages accumulate in Mosquitto. No PASS/FAIL issued. Station holds. ✗ No auto-restart. Mosquitto persisted messages. EC2 restart by SSH — manual intervention required. No process supervisor (systemd / supervisor). No CloudWatch alarm on EC2 health. Average recovery time: 15–45 min depending on engineer availability.
MES Unavailable HIGH Inspection results computed but not written to MES. No inspection records for parts processed during outage. ✗ RabbitMQ held messages in durable queue. But MES adapter Python script had no retry — messages consumed and discarded on write failure in v1.0. Retry logic added in v1.1 patch (3 retries, 30s backoff). But no dead-letter queue — messages still lost after 3 failures. No audit log of lost records.
Ignition Server Down HIGH Entire plant-side pipeline halted. Both Mosquitto and RabbitMQ offline. No image transport, no result routing, no SCADA visibility. ✗ Single point of failure — no redundancy. Mosquitto and RabbitMQ both hosted here. SCADA also down. Root cause of the worst architectural decision. No failover broker. No secondary Ignition server. Planned maintenance required full production stop.
Camera Failure MEDIUM Station goes dark — no images captured. Parts pass through with no quality check. ✓ Python capture script raised exception on camera open failure. MQTT message with error code published. Ignition alarm triggered for station. Alarm was informational only — no automated conveyor stop. Parts could still pass station without inspection until operator responded.
S3 Write Failure MEDIUM Inference result still produced and sent to MES. But inspection record in MES contains broken image URL. ✗ v1.0: boto3 exception silently swallowed — no retry, no alert. Result sent anyway with null image_url field. v1.1 added basic exception logging to EC2 log file. But no automated alert, no MES record correction, no reconciliation job. Broken URLs accumulated silently.
Model Accuracy Drift HIGH PASS/FAIL decisions degrade silently. Defect escapes increase without any system alert. ✗ No automated drift detection. Only identified through weekly manual quality audit comparing MES defect rate vs customer complaints. In one incident (Q3 2021), drift was undetected for ~3 weeks after supplier changed surface finish. ~1,200 parts inspected with degraded model before retraining triggered.
RPi SD Card Failure LOW Station offline until card replaced and OS + software re-imaged. ✓ Spare RPi units kept per line. Ansible playbook for re-imaging (added after first card failure in Oct 2020). Recovery time: ~45 min. No remote health monitoring of SD card wear. No proactive replacement schedule. Failures only discovered reactively when station went offline.

Measured at Station 3, 2021. Total observed: 3.2–4.0 seconds from capture trigger to MES record write.

Full-res capture + resize + blur + equalization + base64 encode

Mosquitto bridge plugin + Internet round-trip to EC2

Biggest consumer — CPU-only EC2, no GPU. SageMaker GPU endpoint → ~200ms.

Engineering Honesty

Known Limitations — Legacy Architecture

An honest assessment of the 2020–21 design decisions and their operational consequences. These limitations directly motivated the next-generation architecture.

Single Point of Failure — Ignition Server
Both Mosquitto and RabbitMQ were hosted on the Ignition Server. Any planned or unplanned server downtime halted the entire inspection pipeline — image transport, result routing, and SCADA visibility all stopped simultaneously. No redundancy or failover.
No OT/IT Security Segmentation
Mosquitto bridged directly to a public AWS EC2 IP with no DMZ layer. Plant network had a direct outbound path to the Internet. Shared username/password auth across all RPi stations — no per-device certificate management.
EC2 Inference — Not Managed or Scalable
ML model ran as a Python Flask/subscriber process on a single EC2 instance. No auto-scaling, no model versioning, no health monitoring. A process crash required manual SSH restart. Model updates meant manual EC2 redeployment and process restart.
No Offline Edge Buffering (v1.0)
Initial deployment had no local buffering on Raspberry Pi. A Wi-Fi dropout caused complete inspection data loss for that window. SQLite offline buffer was added as a patch in v1.1 — not part of the original design specification.
Model Drift Not Monitored
No automated mechanism to detect when model accuracy degraded due to new defect types, lighting changes, or supplier material variation. Drift was only caught through manual quality audits — sometimes weeks after accuracy had already degraded significantly.
MES Integration Tightly Coupled
MES adapter was a Python script with direct stored procedure writes. Any MES schema change or upgrade broke the adapter. No retry logic — failed writes in v1.0 were silently discarded. No dead-letter queue or error tracking.
